# Проверка v2rayN и смены IP

Этот файл нужен, чтобы проверить, что клиент действительно ходит через VPS, а не мимо proxy.

## Что уже считается нормой

Если в логе v2rayN видно такое:

```text
Xray 26.3.27 (Xray, Penetrates Everything.) e9728b4 (go1.26.1 windows/amd64)
```

то клиент уже использует правильный `xray.exe` из нашего репозитория.

Если в логе больше нет строк вида:

```text
REALITY: received real certificate (potential MITM or redirection)
```

то проблема с REALITY-рукопожатием устранена.

## Какой режим использовать сейчас

Пока не использовать `VPN mode` / `TUN mode`.

Использовать только:

- активный сервер VLESS
- режим маршрутизации `Global`
- включенный `System Proxy`

Не использовать одновременно:

- `VPN mode`
- `TUN mode`

Причина: в режиме `VPN mode` у тебя уже были ошибки DNS и таймауты, которые не относятся к серверу REALITY.

## Что нажать в v2rayN

1. Полностью выключить `VPN mode`
2. Полностью выключить `TUN mode`, если он включен отдельно
3. Выбрать рабочий сервер `pupupu`
4. Поставить режим `Global`
5. Включить `Set system proxy`
6. После этого не трогать UI до проверки

Если в логах появляется строка:

```text
Системные прокси изменены - ForcedClear
```

значит системный proxy был снят. В этом состоянии браузер может снова ходить напрямую.

## Как проверить, что proxy реально работает

Открыть PowerShell и выполнить:

```powershell
curl.exe --proxy http://127.0.0.1:10808 https://api.ipify.org
```

Потом:

```powershell
curl.exe --socks5 127.0.0.1:10808 https://api.ipify.org
```

И потом:

```powershell
curl.exe --proxy http://127.0.0.1:10808 https://ipinfo.io/json
```

## Как понимать результат

Если команды показывают IP VPS, например `45.144.30.147`, то proxy работает.

Если команды показывают домашний IP, то трафик идет мимо v2rayN.

Если команды зависают или падают, то проблема в локальном proxy-клиенте v2rayN, а не в VPS.

## Дополнительная проверка локальных портов

В PowerShell выполнить:

```powershell
Get-NetTCPConnection -State Listen | Where-Object { $_.LocalPort -eq 10808 }
```

Нормально, если видны порты:

- `10808` как смешанный порт `mixed port`

В твоей текущей конфигурации v2rayN использует один смешанный порт `10808`, который должен принимать и `SOCKS`, и `HTTP proxy`.

Если его нет, значит локальный proxy-клиент не поднялся как надо.

Если `curl.exe --proxy http://127.0.0.1:10808 ...` возвращает ошибку:

```text
Failed to connect to 127.0.0.1 port 10808
```

то это значит, что смешанный локальный proxy на `10808` не слушает.

В таком случае проверить по порядку:

1. Выполнить `Get-NetTCPConnection -State Listen | Where-Object { $_.LocalPort -eq 10808 }`
2. Открыть настройки v2rayN и проверить, что `Смешанный порт` равен `10808`
3. Убедиться, что после сохранения настроек v2rayN был перезапущен полностью
4. Использовать для проверки именно порт `10808` и для `SOCKS`, и для `HTTP proxy`
5. Сразу проверить SOCKS напрямую:

```powershell
curl.exe --socks5 127.0.0.1:10808 https://api.ipify.org
```

Если SOCKS и HTTP на `10808` не работают, значит проблема не в VPS и не в REALITY, а в локальном proxy-клиенте v2rayN.

Если команда

```powershell
curl.exe --socks5 127.0.0.1:10808 https://api.ipify.org
```

не отвечает вообще, то это означает одно из двух:

1. порт `10808` не слушает
2. порт `10808` слушает, но запрос зависает уже после входа в локальный SOCKS, то есть проблема в локальном клиенте v2rayN или в его outbound-цепочке

В этом случае проверить так:

```powershell
Get-NetTCPConnection -State Listen | Where-Object { $_.LocalPort -eq 10808 }
```

Потом выполнить SOCKS-запрос с таймаутом:

```powershell
curl.exe --connect-timeout 5 --max-time 15 --socks5 127.0.0.1:10808 https://api.ipify.org
```

Если и это зависает или падает, то:

- сервер REALITY уже не главный подозреваемый
- надо смотреть локальные inbound/outbound настройки v2rayN
- особенно важно, не остались ли включенными `TUN`, `VPN mode`, `bypass`, `custom routing` или нестандартные локальные порты

В таком случае следующий обязательный тест это не `curl`, а проверка локально слушающих портов и полный свежий лог v2rayN после чистого перезапуска клиента.

Если команда возвращает:

```text
curl: (28) Connection timed out after 5003 milliseconds
```

то это не похоже на ошибку сервера REALITY.

Это означает, что локальный запрос через SOCKS не смог нормально пройти в разумное время. На практике дальше надо проверять только это:

1. Слушает ли вообще `10808`
2. Не стоит ли в v2rayN другой SOCKS port
3. Не остались ли включенными `VPN mode`, `TUN mode` или пользовательские правила маршрутизации
4. Не снялся ли локальный proxy после переключений в UI

Следующие команды для проверки:

```powershell
Get-NetTCPConnection -State Listen | Where-Object { $_.LocalPort -in 10808,10809 }
```

```powershell
netstat -ano | findstr :10808
```

```powershell
curl.exe --connect-timeout 5 --max-time 15 --socks5 127.0.0.1:10808 http://1.1.1.1
```

Последняя команда нужна, чтобы убрать DNS из проверки и понять, проходит ли вообще простой трафик через локальный SOCKS.

## Если локальный порт принимает соединения, но `curl` все равно висит

Если в логах v2rayN видно строки вида:

```text
accepted tcp:www.google.com:443 [socks -> proxy]
```

но `curl` через `127.0.0.1:10808` все равно уходит в timeout, тогда надо проверять уже не только клиент, но и исходящий интернет самого VPS.

На сервере выполнить:

```bash
curl -4 --max-time 10 https://api.ipify.org
```

```bash
curl -I --max-time 10 https://www.google.com
```

```bash
curl -I --max-time 10 https://2ip.ru
```

Если эти команды на VPS не работают или висят, значит проблема уже не в v2rayN, а в исходящем доступе самого сервера.

Если они работают быстро, а локальный `curl` через `127.0.0.1:10808` все равно висит, тогда проблема остается в клиентской стороне v2rayN.

## Что НЕ означает успех

Строка в логе вида:

```text
accepted tcp:www.google.com:443 [socks -> proxy]
```

не доказывает, что сайт реально открылся через VPS.

Она означает только то, что локальный клиент принял соединение от приложения.

Поэтому главный тест это не лог, а `curl.exe` через `127.0.0.1:10808` и `127.0.0.1:10809`.

## Если IP не меняется

Проверить по порядку:

1. Точно ли выбран сервер `pupupu`
2. Точно ли включен `Set system proxy`
3. Точно ли выключен `VPN mode`
4. Не появлялся ли в логе `ForcedClear`
5. Слушает ли смешанный порт `10808`
6. Что показывает `curl.exe --proxy http://127.0.0.1:10808 https://api.ipify.org`

## Если понадобится режим VPN позже

Сначала добиться стабильной работы обычного `System Proxy`.

Только после этого возвращаться к `VPN mode`, потому что там отдельно придется разбирать DNS внутри TUN.

## Как включить именно режим VPN/TUN

Если цель именно такая: на домашнем ПК включить v2rayN и прогонять через VPS весь трафик как через VPN, то делать надо так.

### 1. Что оставить на вкладке базовых настроек

На вкладке `Ядро: базовые настройки` оставить:

- `Смешанный порт` = `10808`
- `Включить UDP` = `On`
- `Включить сниффинг` = `On`
- в типе сниффинга оставить только `http` и `tls`
- `Включить мультиплексирование Mux` = `Off`
- `Включить фрагментацию (Fragment)` = `Off`

Остальное на этой вкладке не трогать.

### 2. Что включить на вкладке режима TUN

Перейти на вкладку `Настройки режима TUN` и включить только базовую схему:

- включить `TUN mode` / `VPN mode`
- включить `Auto route`, если такой переключатель есть
- включить `Strict route`, если такой переключатель есть
- стек оставить по умолчанию; если есть выбор между несколькими стеками и один из них не работает, пробовать `System`, а потом `gVisor`

### 3. Что указать для DNS в TUN

Если на вкладке TUN есть поля DNS, поставить:

- основной DNS: `1.1.1.1`
- запасной DNS: `8.8.8.8`

Если есть поле для `DoH`, можно поставить:

- `https://1.1.1.1/dns-query`

Если есть `FakeDNS`, пока не включать.

### 4. Что выключить, чтобы не мешало

Перед тестом режима VPN выключить:

- `Set system proxy`
- любые пользовательские routing rules, если они были добавлены вручную
- `bypass` / `direct` правила, если они включались вручную

Для режима `VPN/TUN` системный proxy не нужен.

### 5. Как перезапустить после изменения TUN

После смены настроек:

1. нажать `Подтвердить`
2. полностью закрыть v2rayN, включая иконку в трее
3. запустить v2rayN снова
4. выбрать сервер `pupupu`
5. включить только `VPN mode` / `TUN mode`
6. `Global` можно оставить, но для TUN он уже не главный переключатель

### 6. Как проверить режим VPN

После включения TUN проверить в PowerShell:

```powershell
curl.exe https://api.ipify.org
```

И потом:

```powershell
curl.exe https://ipinfo.io/json
```

Если режим VPN реально поднялся, эти команды должны показывать IP VPS, а не домашний IP.

### 7. Если после включения TUN снова идут DNS timeout

Если снова появляются ошибки вида:

```text
dns: exchange failed
```

то это почти всегда значит одно из двух:

1. у TUN неправильно выставлены DNS
2. выбран неудачный стек TUN

В таком случае менять только по одному параметру:

1. сначала выставить DNS `1.1.1.1` и `8.8.8.8`
2. если не помогло, сменить стек TUN
3. если не помогло, временно выключить `Strict route`

Не менять всё сразу.
